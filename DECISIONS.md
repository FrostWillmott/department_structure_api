# Технические решения

Неочевидные архитектурные решения, принятые в проекте, с обоснованием.

---

## Общий подход

### Чистая архитектура без избыточных абстракций

Распространённая ошибка в тестовых заданиях — добавлять абстракции ради самих абстракций: repository-интерфейсы, service-протоколы, обобщённые базовые классы, фабрики. Это демонстрирует знакомство с паттернами, но сразу возникает вопрос: понимает ли автор, *когда* их применять. Правило, которое здесь используется: добавлять слой только тогда, когда он решает конкретную проблему, видимую в текущей кодовой базе. У каждого существующего слоя (router → service → ORM) есть чёткое обоснование; всё рассмотренное и отброшенное (repository, generic CRUD mixin) исключено, потому что проект пока в этом не нуждается.

## Архитектура

### Нет слоя репозитория

Сервисы обращаются к `AsyncSession` SQLAlchemy напрямую. Слой репозитория добавил бы границу абстракции между сервисной логикой и ORM — это полезно, когда нужно менять хранилище или изолировать SQL от бизнес-логики в больших кодовых базах. Здесь хранилище фиксировано (PostgreSQL), сервисный слой уже тонкий, а лишнее опосредование только запутало бы код.

### Доменные исключения, HTTP-контекст только в роутерах

Сервисы бросают типизированные доменные исключения (`DepartmentNotFoundError`, `CycleDetectedError` и т.д.). Роутеры перехватывают их и конвертируют в `HTTPException`. Это означает, что сервисные функции тестируемы без HTTP-контекста, а маппинг бизнес-ошибки в статус-код находится в одном месте (роутере), а не разбросан по бизнес-логике.

## База данных / ORM

### Уникальность имён контролируется на уровне сервиса, а не через DB constraint

Индекс `UNIQUE(name, parent_id)` выглядел бы естественно, но NULL-значения его ломают: SQL считает `NULL != NULL`, поэтому два корневых отдела (`parent_id IS NULL`) с одинаковым именем не будут пойманы уникальным индексом. Сервис явно делает запрос `WHERE name = ? AND parent_id IS NULL` (или `= ?`), что единообразно обрабатывает оба случая.

### `ON DELETE CASCADE` на уровне БД + `passive_deletes=True` в ORM

Каскадное удаление дочерних отделов и сотрудников объявлено на FK-ограничении (`ondelete="CASCADE"`), а не как ORM cascade. Без `passive_deletes=True` SQLAlchemy не знает, что БД сама обработает каскады — он выдаёт `UPDATE employees SET department_id = NULL` перед `DELETE`, что нарушает ограничение `NOT NULL` на `department_id`. С `passive_deletes=True` SQLAlchemy доверяет БД и отправляет только `DELETE`.

### BFS для обнаружения циклов вместо рекурсивного CTE

Перед разрешением изменения `parent_id` обходим существующее поддерево через BFS (последовательные раунды `SELECT id WHERE parent_id IN (...)`) и собираем всех потомков, затем проверяем, нет ли нового родителя среди них. Альтернатива — один `WITH RECURSIVE` CTE — была бы эффективнее для глубоких деревьев, но добавляет SQL-сложность. Для типичных оргструктур (глубина < 10, сотни узлов) лишние round trip несущественны, а Python-код проще читается и тестируется.

### Режим reassign перемещает только прямых сотрудников

`DELETE ?mode=reassign` перемещает *прямых* сотрудников отдела в целевой, затем удаляет отдел. Дочерние поддеревья со своими сотрудниками всё равно каскадно удаляются БД. Рекурсивное переназначение всего поддерева рассматривалось и было отклонено: массовое переназначение поддерева — это отдельная операция («реорганизация»), которая заслуживает собственного явного эндпоинта, а не флага на удалении.

### Нет eager/lazy loading — явный `selectinload` в точке запроса

Async SQLAlchemy не поддерживает lazy loading (бросает `MissingGreenlet`). Все загрузки связанных объектов используют `selectinload` непосредственно в том запросе, которому они нужны, — поэтому всегда понятно, какой SQL выполняется и когда. У связей в модели нет параметра `lazy=`, чтобы случайно не включить неявную загрузку.

## Тестовая инфраструктура

### Фикстура `db_engine` в function scope (создание/удаление таблиц для каждого теста)

Session-scoped engine был бы быстрее, но pytest-asyncio создаёт отдельный event loop для каждой тестовой функции. Engine, созданный в одном event loop, нельзя безопасно использовать из другого — пул соединений asyncpg привязан к тому loop, в котором был создан. Создание/удаление таблиц per-test добавляет несколько миллисекунд на тест и полностью исключает этот класс сбоев.

### `from main import app` на уровне модуля в conftest.py

`Base.metadata` знает только о таблицах, чьи модельные классы уже импортированы. `conftest.py` импортирует `from app.database import Base`, но не `app.models`. Если `from main import app` отложить внутрь фикстуры `client` (которая выполняется *после* `db_engine`), то `Base.metadata.create_all` запустится на пустом реестре и не создаст ни одной таблицы — первый тест провалится с `UndefinedTableError`, второй пройдёт, потому что к тому моменту `app.models` уже импортирован. Перенос импорта на уровень модуля гарантирует, что модели зарегистрированы до выполнения любой фикстуры.

## Docker

### Многоэтапный Dockerfile (`base` → `production` / `test`)

Этап `production` копирует только `app/`, `alembic/`, `alembic.ini` и `main.py` — без тестов и dev-зависимостей. Этап `test` (`COPY . .`) включает всё. Одноэтапный образ с тестовыми файлами и dev-зависимостями в production был отклонён: он раскрывает внутренний инструментарий и раздувает размер образа.

### Docker Compose profiles для изоляции тестов

Сервисы `db_test` и `test` включены в профиль `test`. `docker compose up` запускает app-стек; `docker compose --profile test up` запускает тесты. Это идиоматический паттерн Compose для опциональных групп сервисов — избавляет от необходимости поддерживать отдельные compose-файлы (`docker-compose.prod.yml`, `docker-compose.test.yml`), которые со временем расходятся.
